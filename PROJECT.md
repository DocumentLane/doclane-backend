# PDF Reader Project Draft

## 1. 프로젝트 기본

### 목표

웹 기반 PDF 리더를 구축한다. 초기 범위는 다음 기능에 한정한다.

- PDF 업로드
- PDF 열람
- 북마크
- 메모

OCR은 기본 기능이 아니라 선택적 후처리 기능으로 둔다.

### 핵심 원칙

- 원본 PDF는 항상 보존한다.
- PDF 렌더링은 `pdf.js` 기반으로 처리한다.
- 문서 조회 성능은 페이지 단위 lazy loading과 range request 기반 로딩으로 확보한다.
- CPU/메모리를 많이 사용하는 작업은 메인 서버에서 처리하지 않는다.
- 스토리지는 private로 운영하고, 문서 열람 시 signed URL을 발급한다.

### 시스템 구성

- 프론트엔드 웹 리더
- 메인 API 서버
- 원격 워커(Mac mini)
- 공용 DB
- 공용 큐(Redis/BullMQ)
- 공용 오브젝트 스토리지(S3 호환)

### 문서 처리 기본 흐름

1. 사용자가 PDF를 업로드한다.
2. 메인 서버가 문서 레코드를 생성하고 원본 PDF를 스토리지에 저장한다.
3. 메인 서버가 워커 큐에 작업을 등록한다.
4. 워커가 문서 메타데이터 또는 OCR 처리를 수행한다.
5. 프론트는 메인 서버에서 문서 메타데이터와 signed URL을 받아 `pdf.js`로 문서를 렌더링한다.

### 문서 열람 방식

- 프론트는 먼저 문서 메타데이터를 API로 조회한다.
- 메인 서버는 권한 확인 후 signed URL을 발급한다.
- 프론트는 signed URL을 `pdf.js`에 전달한다.
- `pdf.js`는 스토리지에 직접 range request를 보내 필요한 바이트만 읽는다.
- 현재 화면에 보이는 페이지와 인접 페이지 중심으로만 렌더링한다.

### OCR 도입 기준

- 텍스트 레이어가 존재하는 일반 PDF는 OCR 없이 처리한다.
- 텍스트 추출이 불가능한 스캔 PDF만 OCR 대상이 된다.
- OCR은 문서 열람 파이프라인과 분리된 비동기 후처리로 둔다.

## 2. 워커

### 역할

워커는 메인 서버와 분리된 비동기 처리 프로세스다. 메인 서버의 부하를 줄이기 위해 무거운 PDF 처리 작업을 전담한다. 워커는 원격 Mac mini에서 실행한다.

워커의 책임은 다음과 같다.

- PDF 메타데이터 추출
- 페이지 정보 추출
- 썸네일/미리보기 생성
- OCR 수행
- 처리 결과를 DB와 스토리지에 기록

### 워커 분리 구성

#### 1) `pdf-metadata`

문서를 빠르게 열 수 있는 최소 상태를 만드는 워커다.

주요 작업:

- 페이지 수 추출
- 페이지별 width/height/rotation 추출
- 텍스트 레이어 존재 여부 판정
- 첫 페이지 썸네일 생성

#### 2) `pdf-page-derivative`

조회 성능과 탐색 UX 개선용 파생 산출물을 만드는 워커다.

주요 작업:

- 페이지 썸네일 생성
- preview image 생성
- 초기 페이지 선처리
- 조회 최적화용 파생 파일 생성

#### 3) `pdf-ocr`

텍스트 레이어가 없는 문서에 OCR을 수행하는 워커다.

주요 작업:

- 페이지 이미지 렌더링
- OCR 수행
- 페이지별 OCR 텍스트 저장
- 검색 인덱스용 텍스트 후처리

### 큐 설계

워커 큐는 역할별로 분리한다.

- `pdf-metadata`
- `pdf-page-derivative`
- `pdf-ocr`

분리 이유:

- 메타데이터 추출은 빠르게 끝나야 한다.
- 파생 이미지 생성은 선택적이지만 조회 UX에 영향을 준다.
- OCR은 가장 무겁고 적체 가능성이 높다.

OCR 큐를 별도 분리해야 일반 문서 준비 작업이 막히지 않는다.

### 상태 공유 방식

워커는 메인 서버에 직접 HTTP로 상태를 보고하지 않는다. 메인 서버와 워커는 공용 DB를 통해 상태를 공유한다.

상태 저장 단위:

- `documents`: 사용자 관점 문서 상태
- `document_jobs`: 작업 상태 요약
- `document_job_logs`: 작업 이벤트 로그

작업 전달 단위:

- 큐: Redis/BullMQ
- 파일 저장소: Object Storage
- 상태 저장소: Postgres

### 워커 처리 흐름

1. 메인 서버가 문서 업로드 완료 후 큐에 작업을 등록한다.
2. 워커가 큐에서 작업을 가져온다.
3. 워커가 스토리지에서 원본 PDF를 읽는다.
4. 워커가 메타데이터, 파생 파일, OCR 결과를 생성한다.
5. 워커가 결과를 DB와 스토리지에 저장한다.
6. 메인 서버는 DB 상태를 조회해 사용자에게 진행 상태를 제공한다.

### 운영 원칙

- 메인 서버는 PDF 파싱과 OCR을 수행하지 않는다.
- 워커는 스토리지, DB, 큐만 직접 사용한다.
- 워커 작업은 재시도 가능해야 한다.
- OCR 실패가 문서 열람 실패가 되면 안 된다.
- 무거운 작업일수록 작은 배치 단위로 나눠 실행한다.

## 3. 메인 서버

### 역할

메인 서버는 PDF 처리 서버가 아니라 제어 서버다. 문서 처리 자체는 워커에 맡기고, 메인 서버는 인증, 권한, 상태 관리, API 제공에 집중한다.

메인 서버의 책임은 다음과 같다.

- 업로드 세션 발급
- 문서 레코드 생성
- 문서 권한 검사
- 문서 메타데이터 조회 API 제공
- signed URL 발급
- 북마크/메모 CRUD API 제공
- 워커 작업 enqueue
- 문서 상태 및 job 상태 조회 API 제공

### 처리 원칙

- PDF 바이너리를 직접 파싱하지 않는다.
- OCR을 수행하지 않는다.
- 업로드 완료 후 워커 큐에 작업만 등록한다.
- 문서 처리 상태는 공용 DB를 조회해 응답한다.

### 업로드 흐름

1. 사용자가 업로드 요청을 시작한다.
2. 메인 서버가 문서 ID와 업로드 세션을 생성한다.
3. 원본 PDF는 private storage에 저장된다.
4. 업로드 완료 후 메인 서버가 워커 큐에 metadata job을 등록한다.
5. 문서 상태는 `uploaded` 또는 `metadata_processing`으로 갱신된다.

### 문서 열람 API 역할

- 프론트가 문서 상세를 조회하면 메인 서버는 문서 메타데이터를 반환한다.
- 프론트가 실제 열람을 요청하면 메인 서버는 권한을 확인한 후 signed URL을 발급한다.
- 프론트는 발급된 signed URL을 사용해 스토리지에서 직접 PDF를 읽는다.

이 구조를 사용하는 이유는 다음과 같다.

- 메인 서버가 대용량 PDF 트래픽을 직접 처리하지 않아도 된다.
- 문서는 private storage로 보호할 수 있다.
- `pdf.js`가 range request를 통해 필요한 바이트만 가져올 수 있다.

### 권장 API 범위

- `POST /documents/upload-session`
- `POST /documents`
- `GET /documents/:id`
- `GET /documents/:id/view`
- `GET /documents/:id/status`
- `GET /documents/:id/bookmarks`
- `POST /documents/:id/bookmarks`
- `GET /documents/:id/notes`
- `POST /documents/:id/notes`

### 상태 관리

메인 서버는 두 종류의 상태를 구분해서 관리한다.

- 문서 상태: 사용자가 보는 준비 상태
- job 상태: 내부 워커 처리 상태

권장 저장 단위:

- `documents`
- `document_jobs`
- `document_job_logs`

### 운영 원칙

- 메인 서버는 가능한 한 stateless 하게 유지한다.
- signed URL은 짧은 만료 시간으로 발급한다.
- 문서 권한 검사는 signed URL 발급 전에 반드시 수행한다.
- 워커와 상태 동기화는 HTTP 호출이 아니라 공용 DB 기반으로 처리한다.

## 4. 프론트

### 역할

프론트는 웹 PDF 리더 UI를 제공한다. 문서 목록, 문서 상세, PDF 리더, 북마크, 메모, 상태 표시를 담당한다.

프론트의 책임은 다음과 같다.

- 문서 메타데이터 조회
- PDF 리더 렌더링
- 북마크 추가/조회
- 메모 추가/조회
- 문서 상태 표시
- signed URL 기반 문서 열람

### 문서 로딩 흐름

1. 프론트가 메인 서버에 문서 메타데이터를 요청한다.
2. 메인 서버가 문서 정보와 signed URL을 반환한다.
3. 프론트가 `pdf.js`에 signed URL을 전달한다.
4. `pdf.js`가 스토리지에 range request를 보내 필요한 PDF 데이터를 읽는다.
5. 현재 화면에 보이는 페이지와 인접 페이지 중심으로 렌더링한다.

### 렌더링 원칙

- `pdf.js`를 사용한다.
- 전체 PDF를 한 번에 모두 렌더링하지 않는다.
- 페이지 단위 lazy render를 적용한다.
- virtualization을 적용해 화면에 보이지 않는 페이지는 DOM/canvas 부담을 줄인다.
- text layer와 canvas layer를 분리해 검색성과 렌더링을 함께 확보한다.
- 확대 시에만 고해상도 렌더링을 수행한다.

### 북마크/메모 처리

북마크와 메모는 PDF 내부에 저장하지 않고 별도 API 데이터로 관리한다.

저장 기준:

- 북마크: `documentId + pageNumber`
- 메모: `documentId + pageNumber + viewport 좌표`

이 방식을 사용하는 이유는 다음과 같다.

- 원본 PDF를 수정하지 않아도 된다.
- 사용자별 데이터 분리가 쉽다.
- 협업 기능으로 확장하기 쉽다.

### 상태 표시

프론트는 문서 상태와 OCR 상태를 구분해 사용자에게 보여준다.

예시 상태:

- 문서 준비 중
- 열람 가능
- OCR 대기 중
- OCR 처리 중
- 처리 실패

OCR이 완료되지 않았더라도 문서 열람 자체는 가능해야 한다.

### 프론트 최적화 원칙

- 문서 메타데이터와 실제 PDF 파일 로딩을 분리한다.
- PDF는 메인 서버를 통해 프록시하지 않고 signed URL로 직접 읽는다.
- 첫 화면 체감 속도를 위해 현재 페이지 중심으로 우선 렌더링한다.
- 썸네일/preview가 준비되어 있으면 보조 UI에 활용한다.
